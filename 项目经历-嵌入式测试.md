# 基于 Qt 和 TCP/IP 的多线程即时通讯系统

> 投递岗位：**嵌入式测试 / 通信协议测试** | 项目类型：独立完成

---

## 项目简介

基于 C++ 和 Qt 开发的 C/S 架构网络通讯系统，采用多线程模型支持多客户端实时通信。本项目从**通信协议、线程安全、异常容错**三个维度完成测试设计与执行，适合嵌入式/通信类测试岗位展示。

- **技术栈**：C++、Qt 5.14、TCP/IP Socket、多线程、自定义应用层协议
- **测试重点**：协议一致性、网络异常容错、并发稳定性、资源生命周期

## 系统架构

```
┌──────────────┐          TCP/IP            ┌──────────────────────┐
│   客户端       │ ◄──────────────────────► │       服务端           │
│              │                            │                      │
│ UI 线程 (主)  │                            │ 主线程  ─ 消息路由    │
│  └ Worker线程 │                            │  ├ QThread-1 ─ sock1 │
│     └ QTcpSocket                          │  ├ QThread-2 ─ sock2 │
│               │                            │  └ QThread-N ─ sockN │
└──────────────┘                            │                      │
                                            │ QMap<用户名,socket>  │
                                            │ (主线程独占，无锁)    │
                                            └──────────────────────┘
```

**线程模型**：
- 服务端：per-client 线程池，主线程监听+路由，工作线程处理 socket I/O
- 客户端：双线程，UI 线程 + Socket Worker 线程，信号/槽跨线程通信

**通信链路**：客户端 Socket → TCP 传输 → 服务端 Socket → 工作线程解析 → 主线程路由 → 目标工作线程写入 → TCP 传输 → 目标客户端

## 通信协议（自定义应用层）

协议基于纯文本帧，`|` 为字段分隔符，首字符识别消息类型：

| 类型码 | 帧格式 | 含义 | 示例 |
|--------|--------|------|------|
| R | `R\|用户名` | 终端注册/上线 | `R\|node_01` |
| S | `S\|目标用户名\|负载` | 点对点单播 | `S\|node_02\|温度:36.5` |
| T | `T\|负载` | 全局组播/广播 | `T\|系统告警:CPU过载` |

**协议设计特点**：
- 定界符为单字符 `|`，解析开销低，适合资源受限场景
- 首字符类型码，单次判断完成路由分发，无需完整解析
- 负载域支持任意内容（含 `|`），使用 `section()` 而非 `split()` 解析

**协议一致性验证**：
- 发送 `S|node_02|hello` → 仅 node_02 收到 `hello`
- 发送 `S|node_02|a|b|c` → node_02 收到 `a|b|c`（负载不被截断）
- 发送 `T|broadcast_msg` → 所有在线节点（除发送者）收到 `broadcast_msg`

## 测试设计与执行

### 一、协议层测试（核心）

| 编号 | 测试项 | 测试方法 | 验证点 |
|------|--------|---------|--------|
| PT-01 | R 帧注册 | 发送 `R\|test_user` → 抓包/日志验证 | 用户表写入、广播上线通知 |
| PT-02 | S 帧单播-正常 | 发送 `S\|target\|msg` → 验证目标端接收 | 目标收到负载、非目标不收到 |
| PT-03 | S 帧单播-目标不存在 | 发送 `S\|ghost\|msg` → 验证错误响应 | 发送端收到"用户不在线" |
| PT-04 | T 帧组播-正常 | 发送 `T\|msg` → 验证所有端接收 | 所有端收到、发送端不收到回环 |
| PT-05 | 帧边界-负载含分隔符 | 发送 `S\|name\|a\|b\|c` | 负载完整性：收到 `a\|b\|c` |
| PT-06 | 帧边界-空负载 | 发送空内容 | 不发送，不崩溃 |
| PT-07 | 帧边界-仅分隔符 | 发送 `T\|` | 群发空字符串，不崩溃 |
| PT-08 | 非法帧格式 | 发送 `X\|data` | 服务端忽略，日志记录 |

### 二、网络异常测试

| 编号 | 测试项 | 模拟方法 | 预期行为 |
|------|--------|---------|---------|
| NT-01 | 连接中断 | 客户端进程直接关闭（不发送断开请求） | 服务端检测到 TCP 断连 → 清理线程 → 广播下线 |
| NT-02 | 频繁重连 | 客户端 10 秒内连→断→连 5 次 | 服务端线程正确创建和回收，无泄漏 |
| NT-03 | 重复注册 | 新客户端使用已在线用户名连接 | 旧连接被踢 → 新连接注册 → 广播更新 |
| NT-04 | 服务端未就绪 | 客户端在服务端启动前发起连接 | 客户端收到连接错误，不崩溃，可重试 |
| NT-05 | 大并发连接 | 10+ 客户端同时发起连接 | 全部注册成功，用户表无遗漏 |
| NT-06 | 长时在线 | 客户端保持连接 30 分钟不发消息 | 连接保持，心跳正常（TCP keepalive） |

### 三、线程安全与并发验证

| 验证项 | 验证方法 | 结果 |
|--------|---------|------|
| 用户表并发安全 | 分析设计：所有 map 操作限定主线程 | 无竞态条件，无死锁风险 |
| 跨线程写 socket | 分析实现：`QMetaObject::invokeMethod` + `QueuedConnection` | 写操作序列化到 socket 所属线程 |
| 线程生命周期 | 动态分析：断连→`deleteLater`→`quit`→`wait`→`delete` | 线程正确回收，无僵尸线程 |

### 四、资源管理验证

| 验证项 | 测试方法 | 结论 |
|--------|---------|------|
| Socket 释放 | 客户端断连后检测服务端 socket 状态 | socket 被正确 close 和 delete |
| 线程回收 | 断连后检测服务端线程数 | 线程数恢复（已回收） |
| 内存泄漏 | 连接→聊天→断连循环 20 次，观察内存 | 无持续增长 |

## 缺陷发现与分析

| 编号 | 严重级别 | 缺陷 | 根因分析 | 修复 |
|------|---------|------|---------|------|
| BUG-01 | P0 致命 | socket 跨线程归属导致通信完全失败 | QTcpServer 创建的 socket 留在主线程，未 moveToThread，工作线程操作非所属线程 socket 导致 I/O 失效 | 将 socket `setParent(nullptr)` + `moveToThread(thread)` 移到工作线程 |
| BUG-02 | P1 严重 | 服务端退出时内存泄漏 | `QThread::finished` 触发时事件循环已退出，`deleteLater` 投递的删除事件无法被执行 | 调整清理时序：先 `deleteLater`（此时事件循环仍在运行），再 `quit→wait→delete thread` |
| BUG-03 | P2 一般 | 协议转发附加调试前缀 | 开发阶段代码残留，透传协议不符合原始设计 | 去除附加前缀，保持协议透明转发 |
| BUG-04 | P2 一般 | 负载含 `|` 时被 `split()` 截断 | `split()` 按全部分隔符切分，未考虑 `section(n)` 可以指定段号 | 改用 `QString::section('|', 2)` 取尾部完整负载 |

## 嵌入式测试的关联性说明

本项目虽然是 PC 端 Qt 程序，但核心技术栈与嵌入式测试高度相关：

| 项目中的技术 | 嵌入式场景对应 |
|-------------|--------------|

| 异常断连处理 | 设备掉线、网络抖动时的容错机制 |
| 线程安全与资源回收 | 嵌入式 RTOS 中的任务间通信与内存管理 |
